Code Review: Implement Redis token store
BOB-412/2 · reviewed by bob-reviewer · 2m ago · APPROVED
✓ Approved
Clean implementation. Token rotation using MULTI/EXEC is the right approach. Connection pool defaults are conservative but reasonable for the current scale. Two minor suggestions below.
B
bob-reviewer
stores/redis.ts
L47-52
suggestion
- const multi = this.redis.multi();
- multi.del(this.key(oldToken));
- multi.set(this.key(newToken), JSON.stringify(session));
+ const pipeline = this.redis.pipeline();
+ pipeline.del(this.key(oldToken));
+ pipeline.setex(this.key(newToken), this.ttl, JSON.stringify(session));
Use pipeline instead of multi here — you don't need atomic guarantees for rotation since the new token is already valid. Pipeline is faster and won't block other commands.

Code Review: Migrate existing sessions
BOB-412/3 · reviewed by bob-reviewer · 5m ago · CHANGES REQUESTED
✕ Changes Requested
Critical: The migration script doesn't handle the case where a session was created after the migration started. Any sessions created during the dual-read window will be orphaned in the old store.
B
bob-reviewer
stores/migrate.ts
L34-41
critical
const oldSessions = await cookieStore.listAll();
for (const session of oldSessions) {
await redisStore.set(session.token, session);
}
This iterates a point-in-time snapshot. Any new sessions created between listAll() and the migration completing will only exist in the cookie store. You need a change-data-capture approach — tail the cookie store for new writes during migration and replay them into Redis.
