## Design decisions (locked)

- **Two-axis selectors.** `[data-theme="bob"][data-mode="dark"] { ... }` for dark Bob; `[data-theme="bob"]:not([data-mode="dark"]) { ... }` for light Bob. Light is the "default" branch — no explicit `data-mode="light"` selector required (jsdom still works because `:not([data-mode="dark"])` matches when the attribute is unset or `"light"`).
- **`data-mode` always set on `<html>`** by `ThemeProvider` (never absent). Avoids ambiguity in CSS selector matching.
- **System mode resolution.** `mode === "system"` resolves to the OS preference at mount time and updates on media-query change. `resolvedMode` always returns `"light"|"dark"` (never `"system"`) — used by callers that need to know what's actually active.
- **No light-OODA design in 6I.** Reuse the dark OODA tokens for `data-theme="ooda"[:not(...)]` so light OODA isn't a hard error. OODA team will design proper light mode later.
- **localStorage keys:** `gmacko-theme` (existing), `gmacko-mode` (new).
- **`prefers-color-scheme` listener cleanup.** `useEffect` returns the removeEventListener cleanup on every mount. Standard React pattern.
- **Token color values.** Use the EXACT hex/oklch from Bob's DESIGN.md verbatim. Don't refactor to a single oklch source-of-truth in this phase — that's a future polish.
- **Satoshi loading is consumer-side.** `theme.css` declares `--font-display` referencing Satoshi; consumer apps add the Fontshare `<link>` themselves. Document in commit message + `@gmacko/ui` README.

### Task 2: `ThemeProvider` refactor — Theme × Mode + system detection

Edit `packages/ui/src/theme-provider.tsx`:
- Add `Mode = "light" | "dark" | "system"` and `ResolvedMode = "light" | "dark"`.
- Props: `defaultTheme`, `defaultMode?: Mode = "system"`.
- State: `theme: Theme`, `mode: Mode`, `resolvedMode: ResolvedMode`.
- Effect: when `mode === "system"`, attach `window.matchMedia("(prefers-color-scheme: dark)")` listener; update `resolvedMode` on change. When `mode === "light"|"dark"`, `resolvedMode = mode`.
- Effect: write `data-theme` AND `data-mode` (= `resolvedMode`) to `<html>`.
- Effect: persist `theme` + `mode` (NOT `resolvedMode`) to localStorage on change.
- Hook returns `{ theme, setTheme, mode, setMode, resolvedMode }`.

Tests — `packages/ui/src/__tests__/theme-provider.test.tsx`. 5 cases:
1. Mounts with `defaultTheme="bob"`, sets `data-theme="bob"` on `<html>`.
2. `defaultMode="light"` sets `data-mode="light"`.
3. `defaultMode="dark"` sets `data-mode="dark"`.
4. `defaultMode="system"` reads `matchMedia` + sets `data-mode` accordingly. Mock `matchMedia` to return dark, assert `data-mode="dark"`.
5. `mode === "system"` listens to media-query change events; trigger a `change` event with `matches: true`, assert `data-mode` flips to `dark`.

Use `@testing-library/react`'s `render` + `act` for state updates.

Commit: `feat(ui): refactor ThemeProvider with mode + system-preference detection`

### Task 3: Token smoke tests via jsdom

**Caveat:** jsdom doesn't fully resolve CSS custom properties from external stylesheets. The token CSS file is in `tooling/tailwind/theme.css`. Two strategies:
- (a) Inject the CSS into the test environment via `import "../../../tooling/tailwind/theme.css?inline"` or fetch the file content + inject via `<style>` tag in the test setup.
- (b) Create a test-fixture CSS file with just the token blocks, import it in the test, and assert against that.

Lean (a) — single source of truth.

4 cases (one per theme×mode combination):
1. `theme="bob", mode="light"` → `--color-accent` matches `#D4850A` (or its oklch).
2. `theme="bob", mode="dark"` → `--color-accent` matches `#E8A33C`.
3. `theme="ooda", mode="light"` → `--color-accent` matches the OODA value.
4. `theme="ooda", mode="dark"` → same.

If jsdom can't read CSS custom-property values from `getComputedStyle` reliably (a known limitation), fall back to a string-content test: parse `theme.css` as text + assert each block contains the right hex value. Less complete but deterministic.
