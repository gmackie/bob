Bob as Shepherd for Hermes Approved · execution update

2026-07-16 repo: bob/bob @ fix/pipeline-visibility author: Claude (Fable)

Overview

Deploy Hermes (Nous Research's self-hosted persistent personal agent, MIT-licensed) on the Bob node, authenticated through our existing model subscriptions via its native OAuth flows. Bob is the shepherd/interface: it hosts the Hermes management surface — first by putting Hermes' own web dashboard behind Bob's auth, then with a Bob-native connectors console built on the dashboard's REST API for managing endpoints (Telegram, Discord, …), cron automations, providers, and health. Telegram comes up first, restoring the daily briefing/wrap-up loop into the Obsidian vault (dead since 2026-04-16).

v3 note: shepherd scope corrected — the primary integration is hosting the Hermes UI / a connector-management wrapper in Bob, not a dispatch lane. Dispatching work items to Hermes over ACP is now a stretch phase. (v2 corrected the identity: Hermes = NousResearch/hermes-agent, a fresh install — nothing is deployed yet, verified.)

Goals & Non-goals

Goals

  • Hermes gateway + dashboard running as systemd services on hetzner-bob (user bob), surviving reboots.
  • Auth via subscriptions: Codex ChatGPT OAuth primary, Grok OAuth fallback. No raw API keys where OAuth works.
  • Hermes dashboard reachable through Bob — routed and gated by Bob's auth, usable from the iPhone.
  • Bob-native connectors console: enable/configure endpoints (Telegram first), allowlists, cron automations, provider status, health — a curated wrapper over the dashboard REST API.
  • Telegram live + daily briefing/wrap-up restored (vault Daily/ note → git push → Telegram summary).
  • Vault unified: one canonical remote, both machines converge.

Non-goals (v1)

  • Dispatching Bob work items to Hermes (ACP lane) — stretch Phase 5, not v1.
  • Endpoints beyond Telegram in v1 — but the console must make adding Discord/Slack/WhatsApp a config-only act.
  • Replacing Bob's claude/grok/cursor lanes.
  • Nous Portal / vLLM / local models (revisit if subscription lanes rate-limit).

Architecture

Telegram (owner DMs) · future: Discord, Slack… ⇄ ┐
                                                  │  hermes gateway (hetzner-bob, systemd, user bob)
                                                  │      ├─ providers: Codex OAuth → Grok OAuth → fallback
                                                  │      ├─ cron: 07:00 briefing · 22:00 wrapup → /opt/obsidian → git → TG
                                                  │      └─ state: ~/.hermes (SQLite memory/skills, auth.json)
        hermes dashboard (FastAPI+SPA, loopback) ─┤
              │ REST /api/config /api/env /api/mcp /api/model cron sessions logs
              │
Bob web/mobile ⇄ Bob app (better-auth) ─┬─ “Hermes” section: connectors console (curated wrapper)
                                        └─ authenticated proxy route → full stock dashboard

Two shepherd layers, cheapest first: (a) proxy — the stock dashboard, bound to loopback, exposed only through an authenticated Bob route (full power, zero UI work, upstream maintains it); (b) wrapper — a Bob-native console for the daily-driver operations (connector on/off, tokens, allowlists, cron toggles, health), consuming the same REST API, in Bob's design language and mobile-friendly. Hermes' endpoints and cron run independently of Bob — the daily loop survives Bob outages.
